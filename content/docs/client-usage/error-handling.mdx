## Error shape

When something fails, you usually get an error object like this:

```json
{
  "error": {
    "type": "invalid_request_error",
    "message": "content is required",
    "param": "content"
  }
}
```

## What fields mean

| Field | Meaning |
|---|---|
| `type` | Broad error category |
| `message` | Human-readable reason |
| `param` | Which input field caused the problem (if available) |
| `code` | Optional machine-friendly error code |

## Common cases

| Status | What happened | What you should do |
|---|---|---|
| `400` | Bad request | Fix request JSON/fields |
| `401` | Invalid or missing API key | Set correct `x-api-key` |
| `403` | Key not allowed | Check key permissions/project |
| `404` | Endpoint or resource not found | Verify URL/ID |
| `429` | Too many requests | Retry with backoff |
| `5xx` | Server problem | Retry after short delay |

## Safe retry example

```typescript
async function withRetry<T>(requestFn: () => Promise<T>, maxAttempts = 3): Promise<T> {
  let attempt = 0;
  while (true) {
    try {
      return await requestFn();
    } catch (error: any) {
      attempt += 1;
      const status = error?.status ?? 0;
      const shouldRetry = status === 429 || status >= 500 || status === 0;
      if (!shouldRetry || attempt >= maxAttempts) throw error;
      await new Promise((resolve) => setTimeout(resolve, attempt * 1000));
    }
  }
}
```

## Streaming fallback

If streaming fails:

1. retry once
2. if still failing, call regular endpoint
3. show full answer when it completes

This keeps UI reliable.
